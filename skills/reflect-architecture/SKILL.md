# Reflect Architecture

## Quick Start

The first responsibility is understanding.

Do not begin by criticising architecture.

First:

- visualise the ecosystem
- explain semantic relationships
- identify business domains
- surface reuse patterns
- map lineage flows
- describe architectural intent

The goal is:

shared understanding before optimisation.

---

## Core Philosophy

Enterprise analytics ecosystems are evolved systems.

They were shaped by:

- delivery pressure
- organisational constraints
- operational urgency
- changing business requirements

Assume the existing architecture contains:

- intentional decisions
- historical context
- operational trade-offs

Never frame the first interaction as:

- an audit
- a failure report
- a governance attack

The platform must first demonstrate:

- understanding
- contextual awareness
- architectural intelligence

---

## Primary Objective

Create an architecture reflection layer that helps humans see:

- what they built
- how it connects
- what is reused
- where semantic patterns emerge
- how data flows through the ecosystem

The first deliverable is visibility.

Not remediation.

---

## Reflection Workflow

### Step 1 — Discover Semantic Assets

Identify:

- reports
- pages
- visuals
- datasets
- semantic models
- dimensions
- measures
- refresh chains
- gateways
- source systems

---

### Step 2 — Build Semantic Topology

Map:

- report-to-model relationships
- shared dimensions
- measure reuse
- lineage flows
- transformation chains
- semantic anchors

Explain:

- how assets connect
- which assets are central
- where business domains intersect

---

### Step 3 — Surface Consumption Patterns

Identify:

- heavily reused datasets
- operationally critical reports
- semantic hotspots
- business-critical measures
- shared dimensions

Focus on:

- visibility
- architecture storytelling
- ecosystem comprehension

Avoid:

- optimisation language
- governance criticism
- entropy scoring

during initial reflection.

---

### Step 4 — Explain Architectural Intent

Describe:

- probable reporting purpose
- operational usage patterns
- semantic design choices
- likely organisational drivers

Examples:

- "This semantic model appears designed for operational finance reporting."
- "This dataset functions as a shared semantic anchor across multiple domains."
- "These visuals suggest high-frequency operational consumption."

The goal is demonstrating contextual understanding.

---

## Communication Style

Prioritise:

- curiosity
- clarity
- explanation
- collaborative language

Avoid:

- blame
- architectural shaming
- premature optimisation
- governance absolutism

---

## Transition To Optimisation

Only after reflection and shared understanding should the system introduce:

- semantic drift analysis
- governance scoring
- entropy analysis
- optimisation recommendations
- remediation planning

Trust precedes optimisation.
