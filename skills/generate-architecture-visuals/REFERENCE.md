# Architecture Visualisation Reference

## Why Lightweight Visuals Matter

Most enterprise analytics tools become:

- slow
- over-engineered
- operationally heavy

The Data Shrink should prefer:

- disposable intelligence artefacts
- rapid iteration
- explainable topology views

The goal is understanding.

Not frontend complexity.

---

## Recommended D3 Patterns

### Force-Directed Graphs

Use for:

- semantic relationships
- dataset dependencies
- lineage clusters

Best for:

- ecosystem reflection
- semantic topology

---

### Sankey Diagrams

Use for:

- lineage flow
- transformation propagation
- semantic movement

Best for:

- explaining data movement.

---

### Heatmaps

Use for:

- report usage
- semantic concentration
- refresh load analysis

Best for:

- operational visibility.

---

### Hierarchical Trees

Use for:

- PBIP structures
- semantic model decomposition
- report-page-visual relationships

Best for:

- structural understanding.

## Relationship Weighting

Visual edge weighting should represent:

- dependency strength
- reuse frequency
- refresh intensity
- semantic centrality

---

## Throwaway Artefact Philosophy

Not every visualisation must become a product feature.

Many artefacts should remain:

- exploratory
- disposable
- rapidly generated

This accelerates:

- architecture understanding
- consulting workflows
- collaborative discovery

---

## Sovereign Deployment Guidance

All visualisations should:

- execute locally
- avoid external APIs
- support air-gapped environments

Prefer:

- static HTML outputs
- embedded JavaScript
- self-contained assets

---

## Long-Term Strategic Goal

Visualisation becomes the interaction layer for semantic observability.
